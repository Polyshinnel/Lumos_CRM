# Шаг 13. Проверка logout-функциональности

**Этап:** 4 — Финализация  
**Тип:** Ручная проверка  
**Статус:** [ ] Не выполнен

---

## Цель

Убедиться, что кнопка выхода в header корректно выполняет logout и перенаправляет пользователя на страницу логина.

---

## Тест-кейсы

### 13.1 Кнопка выхода видна и доступна

- [ ] Кнопка с иконкой `IconLogout` отображается в правой части header
- [ ] При наведении появляется Tooltip с текстом «Выход»
- [ ] Кнопка кликабельна

### 13.2 Процесс logout

При нажатии на кнопку выхода:

- [ ] Вызывается `logout()` из `useAuthStore` — очищается токен и состояние
- [ ] Выполняется `navigate('/login')` — редирект на страницу логина
- [ ] Пользователь видит страницу логина

### 13.3 Состояние после logout

- [ ] Токен удалён из localStorage / Zustand store
- [ ] Попытка перейти на защищённую страницу (например, `/`) перенаправляет на `/login`
- [ ] Данные пользователя (`user`) очищены в store

### 13.4 Серверный logout (заметка)

> **Открытый вопрос:** Функция `logout()` из `authApi.ts` существует, но может не вызываться. При реализации стоит вызывать серверный logout перед очисткой токена, чтобы инвалидировать JWT на сервере.

**Рекомендуемый порядок:**
1. Вызвать серверный `POST /api/auth/logout` (чтобы инвалидировать JWT)
2. Очистить токен и состояние в Zustand store
3. Перенаправить на `/login`

**Обработка ошибок:** Если серверный logout завершился ошибкой (сеть недоступна, токен уже истёк) — всё равно выполнить локальный logout и редирект.

---

## Критерии завершения

- [ ] Кнопка выхода отображается с Tooltip
- [ ] Нажатие выполняет logout и redirect на `/login`
- [ ] Токен и данные пользователя очищены после logout
- [ ] Защищённые маршруты недоступны после logout
